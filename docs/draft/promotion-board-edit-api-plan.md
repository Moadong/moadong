# 홍보 게시판 수정 API 연동 구현 계획

## 배경

현재 백엔드에는 홍보 게시판 조회/생성/수정 API가 이미 구현되어 있다.

- 목록 조회: `GET /api/promotion`
- 생성: `POST /api/promotion`
- 수정: `PUT /api/promotion/{articleId}`

반면 개발자 포털 페이지 `backend/src/main/resources/static/dev/index.html`에는 홍보 게시판 관리 섹션이 아직 없다. 따라서 이번 작업의 핵심은 백엔드 API 추가가 아니라, 개발자 페이지에서 홍보 게시글을 조회하고 수정 API를 호출할 수 있는 UI를 붙이는 것이다.

## 현재 구현 상태

### 백엔드

- `PromotionArticleController`에 홍보 게시글 목록 조회, 생성, 수정 엔드포인트가 존재한다.
- `PromotionArticleService`에서 `articleId`로 게시글을 조회한 뒤 요청값으로 필드를 갱신한다.
- 수정 시 `clubId`로 동아리를 다시 조회하여 `clubName`도 함께 갱신한다.
- `PromotionArticleUpdateRequest`는 아래 필드를 모두 필수로 요구한다.
  - `clubId`
  - `title`
  - `location`
  - `eventStartDate`
  - `eventEndDate`
  - `description`
  - `images`
- 서비스 테스트에 수정 성공 및 게시글 미존재 예외 케이스가 존재한다.

### 프론트

- 개발자 포털에는 현재 `동아리`, `단어사전`, `배너 이미지`, `FCM`, `이미지 변환 배치` 섹션만 있다.
- 동아리 수정은 별도 페이지 `backend/src/main/resources/static/dev/edit.html`에서 상세 조회 후 수정 API를 호출하는 구조로 구현돼 있다.
- 홍보 게시판 관련 섹션, 목록 렌더링, 수정 폼, 수정 요청 로직은 아직 없다.

## 목표

개발자 포털에서 아래 흐름을 지원한다.

1. 홍보 게시글 목록을 조회한다.
2. 수정할 게시글을 선택한다.
3. 기존 값을 폼에 채운다.
4. 수정 내용을 입력한다.
5. `PUT /api/promotion/{articleId}` 호출로 저장한다.
6. 저장 성공 후 목록과 폼 상태를 서버 기준 최신값으로 동기화한다.

## 범위

### 포함

- 개발자 포털 내 홍보 게시판 관리 UI 추가
- 홍보 게시글 목록 조회 기능
- 홍보 게시글 수정 기능
- 로그인 토큰을 이용한 인증 헤더 처리
- 입력값 검증 및 성공/실패 메시지 표시
- 저장 후 목록/선택 상태의 일관성 유지

### 제외

- 일반 사용자용 홍보 게시판 화면 변경
- 백엔드 도메인 구조 변경
- 홍보 게시글 삭제 기능 추가
- 홍보 게시글 상세 조회 API 신규 개발
- 이미지 업로드 전용 API 신규 개발

## 제안 구현 방식

이번 작업은 기존 개발자 포털 패턴에 맞춰 `index.html` 내부에 홍보 게시판 관리 섹션을 추가하는 방향이 가장 자연스럽다.

이유는 다음과 같다.

- 이미 로그인, 토큰, 성공/실패 메시지, 리스트 로딩 패턴이 `index.html`에 모여 있다.
- 목록 조회 후 선택 기반 편집 UI를 같은 페이지 안에서 구현할 수 있다.
- 별도 페이지를 만들지 않아도 수정 대상 선택과 저장 흐름을 빠르게 붙일 수 있다.

단, 폼이 길어져 가독성이 나빠질 경우 `edit.html` 패턴을 재사용한 `promotion-edit.html` 분리도 차선안으로 유지한다.

이번 1차 구현에서는 별도 페이지를 만들지 않고, 개발자 포털 내부 섹션에서 목록과 편집 폼을 함께 제공하는 것을 기본안으로 확정한다.

## 상세 구현 계획

### 1. 네비게이션 및 섹션 추가

`backend/src/main/resources/static/dev/index.html`에 다음을 추가한다.

- 상단 네비게이션에 `홍보 게시판` 링크 추가
- 로그인 후 노출되는 `promotion` 섹션 추가
- 목록 새로고침 버튼
- 선택된 게시글 편집 폼
- 저장 결과 메시지 영역

### 2. 상태 변수 추가

기존 `club`, `dict`, `banner`, `fcm` 섹션과 유사하게 다음 상태를 추가한다.

- 전체 홍보 게시글 목록
- 현재 선택된 게시글 ID
- 현재 선택된 게시글 원본 데이터
- 로딩 상태
- 저장 중 상태
- 목록이 한 번 이상 로드됐는지 여부
- 수정 폼 dirty 여부
- 마지막 저장 성공 메시지 표시 여부

### 3. 목록 조회 구현

버튼 클릭 시 `GET /api/promotion`을 호출한다.

- 응답의 `data.articles`를 기준으로 목록 렌더링
- 제목, 동아리명, 장소, 행사 기간, 이미지 수 정도를 요약 표시
- 클릭 시 해당 게시글을 편집 대상으로 선택
- 선택 시 폼에 기존 값 주입
- 별도 상세 조회 API가 없으므로, 목록 응답의 각 게시글 데이터를 그대로 편집 원본으로 사용

추가 고려사항:

- 목록 API는 공개 엔드포인트지만, 개발자 페이지에서는 로그인 상태에서만 노출하는 기존 패턴을 유지한다.
- 빈 목록일 때 안내 메시지를 보여준다.
- 로그인 직후 자동 로드 여부는 기존 `dict`, `banner`, `fcm` 패턴과 맞춰 `loadPromotionIfVisible()` 형태로 붙인다.

### 4. 수정 폼 구현

수정 폼은 `PromotionArticleUpdateRequest` 구조와 1:1로 맞춘다.

- `clubId`
- `title`
- `location`
- `eventStartDate`
- `eventEndDate`
- `description`
- `images`

입력 방식 제안:

- `clubId`: 읽기 전용 input
- `title`: text input
- `location`: text input
- `eventStartDate`, `eventEndDate`: `datetime-local`
- `description`: textarea
- `images`: 줄바꿈 기반 다중 URL 입력 textarea

이미지 필드는 백엔드에서 `List<String>`를 요구하므로, 저장 직전에 배열로 정규화한다.

추가 결정:

- `clubId`는 1차 구현에서 수정 불가로 둔다.
- 이유는 현재 API가 `clubId` 변경을 허용하지만, 개발자 포털에서 이를 열어두면 게시글 소속 동아리가 의도치 않게 바뀔 수 있기 때문이다.
- 향후 운영 요구가 확인되면 `clubId` 변경 기능을 별도 UX로 분리한다.

폼 하단에는 아래 보조 액션을 둔다.

- 선택 해제
- 원본 값으로 되돌리기

### 5. 데이터 변환 규칙

날짜와 이미지는 프론트에서 명시적인 변환 규칙을 둔다.

- 목록 응답의 `Instant` 문자열은 `datetime-local` 입력값으로 변환한다.
- 저장 시 `datetime-local` 값을 `new Date(value).toISOString()`으로 변환한다.
- 표시 시간대와 저장 시간대 차이를 줄이기 위해, 기존 `edit.html`의 날짜 변환 유틸 패턴을 재사용하거나 동일한 헬퍼를 둔다.
- `images` textarea는 줄바꿈 기준으로 split 하고, trim 후 빈 줄은 제거한다.

검증 규칙:

- `eventStartDate`는 `eventEndDate`보다 늦을 수 없다.
- `images`는 최소 1개 이상이어야 한다.
- 각 이미지 값은 빈 문자열이면 안 된다.

### 6. 수정 API 연동

저장 버튼 클릭 시 아래 순서로 처리한다.

1. 필수값 검증
2. 날짜값을 ISO 문자열로 변환
3. 이미지 입력값을 문자열 배열로 변환
4. `PUT /api/promotion/{articleId}` 호출
5. 성공 시 성공 메시지 표시
6. 현재 목록을 다시 조회해 서버 기준 최신값으로 동기화
7. 같은 `articleId`를 다시 선택 상태로 복원

요청 예시:

```json
{
  "clubId": "65f0c1d5a4b1c92f57d51111",
  "title": "수정된 제목",
  "location": "학생회관 3층",
  "eventStartDate": "2026-04-01T00:00:00.000Z",
  "eventEndDate": "2026-04-10T00:00:00.000Z",
  "description": "수정된 설명",
  "images": [
    "https://example.com/image1.png",
    "https://example.com/image2.png"
  ]
}
```

인증 헤더는 기존 `headers()` 유틸을 그대로 사용한다.

목록 재조회 방식을 우선하는 이유는 다음과 같다.

- 서버가 `clubName` 등 파생값을 다시 계산해 저장할 수 있다.
- 로컬 상태만 부분 갱신하면 서버와 표시값이 어긋날 수 있다.

### 7. 사용자 피드백 및 예외 처리

다음 케이스를 처리한다.

- 로그인하지 않은 경우
- 게시글을 선택하지 않은 상태에서 저장 시도
- 필수값 누락
- 시작일이 종료일보다 늦은 경우
- 이미지 배열이 비어 있는 경우
- 서버가 400을 반환하는 경우
- 서버가 403을 반환하는 경우
- 서버가 404 또는 기타 오류를 반환하는 경우
- 네트워크 오류

UI 피드백 원칙:

- 로딩 중 버튼 비활성화
- 저장 중 버튼 문구 변경
- 성공 시 초록 메시지와 토스트 표시
- 실패 시 에러 메시지와 원인 노출

추가 UX 원칙:

- 저장 중에는 목록 선택 변경과 새로고침 버튼도 함께 비활성화한다.
- 다른 게시글로 이동하려 할 때 dirty 상태면 경고 후 진행한다.

- 현재 선택된 게시글 강조 표시
- 수정 후 정렬 유지

## 검증 계획

### 수동 테스트

- 개발자 계정 로그인 후 홍보 게시판 섹션이 보이는지 확인
- 로그인 후 자동으로 목록이 로드되는지 확인
- 목록 조회 버튼 클릭 시 게시글 목록이 정상 노출되는지 확인
- 특정 게시글 선택 시 폼에 기존 값이 채워지는지 확인
- 제목, 장소, 날짜, 설명, 이미지 URL 수정 후 저장이 되는지 확인
- 저장 후 다시 조회했을 때 변경값이 반영되는지 확인
- 저장 후 같은 게시글이 계속 선택된 상태로 유지되는지 확인
- 잘못된 날짜 범위 입력 시 저장이 차단되는지 확인
- 수정 중 다른 게시글을 클릭할 때 dirty 경고가 정상 동작하는지 확인
- 잘못된 입력일 때 사용자에게 오류 메시지가 보이는지 확인

### 백엔드 회귀 확인

- 기존 `PromotionArticleServiceTest`가 통과하는지 확인
- 필요 시 프론트 변경으로 인한 백엔드 수정은 최소화한다

## 리스크 및 확인 포인트

- 현재 홍보 게시글의 이미지 편집은 URL 배열 기준이다. 개발자 페이지에서 파일 업로드까지 기대한다면 별도 업로드 흐름이 추가로 필요하다.
- 목록 API 응답 구조가 `data.articles`인지 실제 런타임 기준으로 한 번 더 확인하는 것이 안전하다.
- 날짜 입력은 브라우저 로컬 시간대를 사용하므로, `datetime-local`과 `toISOString()` 변환 시 표시와 저장 시점의 시간대 차이를 주의해야 한다.
- 상세 조회 API가 없기 때문에, 목록 응답에 편집에 필요한 모든 필드가 항상 포함된다는 가정이 깨지면 프론트가 바로 영향을 받는다.
- `PUT /api/promotion/{articleId}`는 `/api/admin/**` 하위가 아니므로 보안은 전역 매처가 아니라 메서드 단 `@PreAuthorize`에 의존한다. 변경 시 보안 설정 회귀에 주의가 필요하다.

## 권장 결정

1차 구현은 `index.html` 내부에 `홍보 게시판 관리` 섹션을 추가하는 방식으로 진행한다.

세부 결정은 아래 기준으로 고정한다.

- `clubId`는 읽기 전용으로 둔다.
- `images`는 줄바꿈 기반 입력으로 처리한다.
- 저장 후에는 로컬 상태를 직접 덮어쓰지 않고 목록을 재조회한다.
- 로그인 후 섹션이 보이면 자동으로 홍보 게시글 목록을 로드한다.

이 방식이 가장 빠르고, 기존 개발자 포털 구조와도 잘 맞는다. 구현 중 폼 복잡도가 높다고 판단되면 2차로 별도 편집 페이지 분리를 검토한다.
